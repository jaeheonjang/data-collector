Action 중심 조작 모델

  • 수집: collector 시작/중지를 toggle 버튼으로 제공
  • 수집 모니터링: 기본 1초마다 확인. 파일 10개 증가와 각 파일명 timestamp가 100ms 간격인지 확인해 정상 판정
  • 용량 모니터링: 수집 정상 여부가 아니라 10G 초과 시 주의 알림을 표시하는 기준으로 사용
  • 수집 fatal UX: fatal 발생 시 문제 설명 박스를 띄우고, 사용자가 계속 진행 또는 수집 정지 중 선택
  • 전송: 테스트 이름 입력 후 전송 시작을 누르면 route on, port-forward on, 전송 서버 시작, 다운로드 실행, 전송 서버 중지, route/port-forward 원복을 자동 진행
  • 정리: 산출물 정리는 수집 action이 아니라 전송 성공 후 수행되는 후처리로 분류. 기본값 on인 auto delete toggle로 제어
  • 수집 fatal 처리: collector 자동 중지 없음. 사용자가 상황 확인 후 toggle off 여부를 결정하도록 안내
  • 전송 재시도: 다운로드 실패 시 다운로드 폴더를 밀지 않고 전송 서버와 downloader를 재시작해 최대 3회 자동 재시도. 3회 실패 후에는 정지 또는 재시도를 사용자가 선택
  • 원복 실패: port-forward/route 원복 실패 시 직접 실행할 명령을 메시지 박스로 안내
  • 반복 실행: 테스트 목록을 순회하며 필요한 시점에 수집 toggle과 전송 시작 action을 사용
  • 수동 절차의 터미널 분리는 웹 UI 구조로 끌고 오지 않음
  • 터미널 1/2/3 정보는 버튼 구조가 아니라 각 action의 실행 위치와 로그 맥락으로 표시

프로세스 추적 추천안

  • MVP는 파일 기반 상태 저장 사용: runtime/processes.json, runs/{run_id}/state.json, runs/{run_id}/events.jsonl
  • 유지형 프로세스는 SSH 실행 세션이 살아 있으면 실행 중으로 보고, 세션 종료/끊김/exit status로 상태 판단
  • 보조 확인 예시: 앱 재시작 또는 상태 불일치 시 pgrep, kubectl exec ... pgrep, ip route get 사용
  • 앱 재시작 시 runtime/processes.json을 읽고 필요한 경우에만 collector, port-forward, file server 상태를 재확인
  • 앱이 만들지 않은 기존 프로세스가 감지되면 외부 프로세스 감지 및 수동 확인 필요 상태로 표시

  • 실제 내부망에서는 해당 위치에 실제 파일을 넣는 방식으로 운영

환경 설정 예시

  • MVP에서는 별도 admin token 없이 내부망 접근 전제로 시작
  • 필수 env: COLLECTOR_HOST, COLLECTOR_USER, COLLECTOR_PASSWORD, COLLECTOR_SUDO_PASSWORD, COLLECTOR_POD_NAME_PATTERN
  • 다운로드 env: DOWNLOAD_HOST, DOWNLOAD_USER, DOWNLOAD_PASSWORD, DOWNLOAD_ROOT_DIR
  • SSH/sudo password는 .env에서 읽고 화면/로그에 출력하지 않음
  • password, full SSH URL, 민감 경로는 로그 마스킹
  • backend는 allowlist action만 실행하고 임의 shell command 입력은 허용하지 않음

확정된 세부 정책

  • SSH 상태 매핑: 연결 중 running, exit code 0 completed, exit code != 0 failed, 연결 끊김 needs_check, stderr+0 completed_with_warning
  • auto delete 실패: 실패 원인 출력, 재시도/취소 버튼 제공. 취소 시 수동 삭제 안내
  • 원복 실패: 직접 실행할 port-forward/route 원복 명령과 상태 재확인 버튼 표시
  • 스크립트 배포: repo의 scripts/transfer/ 파일을 항상 overwrite. 백업은 MVP 범위에서 제외

  • 로그/상태 저장: JSONL 로그, run 상태 파일, 추후 SQLite 전환 여부 결정

결정 우선순위

  • 1순위: 테스트 입력 모델과 실행 단위. UI 구조와 API 경계가 여기서 결정됨
  • 2순위: 프로세스 제어와 실패 정책. 내부망 테스트 중 복구 가능성을 좌우함
  • 3순위: 스크립트 배포와 로그 저장. 구현 방식은 중요하지만 MVP에서 단순화 가능
  • 4순위: 보안/접근. 내부망 단일 사용자 전제면 최소 토큰 보호로 시작 가능

MVP 기능 초안

  • 수집 패널: collector toggle, 파일 생성 정상 여부, 파일 수, 용량 표시
  • 수집 fatal: 1초 체크에서 파일 10개 증가 또는 100ms timestamp 간격이 한 번이라도 어긋나면 fatal error 상태 표시
  • 수집 fatal 선택 박스: 문제 설명, 계속 진행, 수집 정지 버튼 제공
  • 수집 주의: 용량이 10G를 초과하면 주의 알림 표시
  • 전송 패널: 테스트 이름 입력, 전송 시작 버튼, 자동 단계 진행 상태 표시
  • 전송 옵션: auto delete toggle 제공. 기본 on, off면 전송 성공 후 수동 삭제 버튼 제공
  • 전송 자동 단계: route on -> port-forward on -> server on -> download -> server off -> auto delete 조건 확인 -> port-forward/route off
  • 전송 retry: 다운로드 실패 시 server 재기동 후 최대 3회 자동 재시도. 3회 실패 후 정지 또는 다시 시도 선택 박스 표시
  • 원복 실패 안내: port-forward/route 직접 실행 명령을 메시지 박스로 표시
  • One-call 실행: 테스트 목록을 순회하며 수집 상태 확인 후 전송 시작 action을 자동 실행
  • One-call 종료/원복: 실행 중인 유지형 프로세스 종료, route 원복, 최종 상태 확인
  • 로그 출력: 각 명령의 stdout/stderr, 시작/종료 시각, exit code, 실행 서버, 실행 단계 표시

사용자 확인 필요

  • MVP에서 로그 저장을 파일 기반으로 시작할지, SQLite까지 포함할지 결정
  • 수집 시작/서버 시작 명령은 dpp pod 내부에서 실행하는 것으로 보이므로, 웹앱이 kubectl exec로 실행해도 되는지 확인
  • 수집 스크립트 실행 인자 형식과 성공/실패 판정 기준 확인
  • sudo password env 저장 허용 범위와 내부망 보안 정책 확인
  • 웹앱 접근 사용자가 1명인지, 간단한 로그인/토큰 보호가 필요한지 확인
  • downloader가 부분 다운로드 파일을 남기는지, 이어받기/덮어쓰기/검증 로직이 있는지 확인
  • 가이드의 --rout-dir 옵션명이 실제 스크립트에서 맞는지 확인
  • repo에 실제 server/downloader 스크립트를 넣을 수 있는지, 아니면 dummy placeholder로 개발 후 내부망에서 교체할지 결정
