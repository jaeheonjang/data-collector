Kubernetes Pod A /dev/shm 접근 방식 정리

기존 Pod A를 수정하지 않고, Pod B에서 /proc/<PID>/root/dev/shm 경로를 통해 링버퍼 데이터를 읽는 2가지 방식 비교

작성일: 2026-06-12 관점: 성능 / 운영 / 보안 대상: Pod A shm ringbuffer → Pod B consumer

목차

1. 핵심요약 2. 배경과 전제 3. 방식 1: privileged Pod B 동적 접근 4. 방식 2: PID 주입 hostPath mount 5. 두 방식 비교 6. 보안 확인 항목 7. 추천 진행안 8. Codex 요청용 요약

1핵심요약

/proc/${PID}/root/dev/shm 접근 방식은 크게 2가지입니다.

구분 방식 핵심 개념 추천 용도
방식 1 Pod B가 privileged로 /host/proc를 마운트하고 PID를 동적으로 찾아 접근 Pod B가 직접 Pod A PID를 찾고 /host/proc/<PID>/root/dev/shm을 읽음 현재 채택 방향 / 자동화 운영
방식 2 외부에서 Pod A PID를 확인한 뒤 /proc/<PID>/root/dev/shm을 Pod B에 hostPath로 직접 mount Pod B는 이미 mount된 /pod-a-shm만 읽음 개념 후보였으나 실제 mount 실패로 보류

방식 1 한 줄 정리

Pod B가 직접 PID를 찾으므로 자동화가 가능하지만, privileged, hostPID, host /proc mount가 필요할 가능성이 큽니다.

방식 2 한 줄 정리

외부에서 PID를 찾아 Pod B에 mount하면 권한을 줄일 수 있을 것으로 보였지만, 실제 테스트에서 runc invalid argument 오류가 발생해 현재는 보류합니다.

중요: 두 방식 모두 같은 Node에서만 유효합니다. /proc/<PID>는 Pod A가 실행 중인 Node 안에서만 존재합니다.

2배경과 전제

현재 상황

  • 기존 Helm chart로 여러 Pod가 배포되어 있음
  • 그중 Pod A가 /dev/shm에 링버퍼 데이터를 기록 중
  • 기존에는 Pod A 내부 consumer가 100ms마다 링버퍼를 읽어 약 700KB 크기의 header+data bin 파일을 작성
  • 새 구조에서는 Pod A를 수정하지 않고 별도 Pod B를 추가하여 consumer 역할을 수행하고자 함

성능 전제

항목 값 / 판단
데이터 크기 약 700KB / 100ms
초당 처리량 약 7MB/s, 약 56Mbps
네트워크 필요성 두 방식 모두 네트워크를 사용하지 않음
shared memory 장점 /dev/shm이 tmpfs 기반이라면 대부분 유지
zero-copy 여부 완전한 zero-copy는 아님. Pod B가 읽을 때 user-space copy는 발생
정리: /proc/<PID>/root/dev/shm/<file>은 별도 복사본이 아니라 Pod A 컨테이너가 보는 /dev/shm 객체를 다른 경로로 바라보는 방식입니다.

/proc/<PID>/root/dev/shm으로 접근되는 이유

Linux에서 컨테이너는 별도의 머신이 아니라, host kernel 위에서 namespace와 cgroup으로 격리된 프로세스 집합입니다. Kubernetes Pod 안의 각 컨테이너도 결국 Node 입장에서는 하나 이상의 Linux 프로세스로 실행되며, 그 프로세스마다 host 기준 PID가 존재합니다.

host의 /proc/<PID> 디렉터리는 해당 프로세스에 대한 kernel view입니다. 여기서 /proc/<PID>/root는 그 프로세스가 자신의 root filesystem으로 보고 있는 위치를 가리키는 특수 symlink입니다. 즉 host에서 /proc/<PID>/root/etc를 보면, 해당 컨테이너 내부에서 /etc로 보이는 경로를 바깥에서 따라 들어가는 것과 같습니다.

같은 원리로 Pod A 컨테이너 내부의 /dev/shm은 host에서 /proc/<PodA_PID>/root/dev/shm 경로로 접근할 수 있습니다. 이 경로는 파일을 복사해서 만든 별도 디렉터리가 아니라, Pod A 프로세스의 mount namespace와 root filesystem 관점에서 /dev/shm을 다시 찾아 들어가는 우회 경로입니다.

경로 의미
/proc host kernel이 제공하는 프로세스 정보 파일시스템
/proc/<PID> host에서 본 특정 프로세스의 상태, namespace, fd, mount 정보
/proc/<PID>/root 그 프로세스가 컨테이너 내부에서 /로 보고 있는 root filesystem
/proc/<PID>/root/dev/shm 그 프로세스가 컨테이너 내부에서 /dev/shm으로 보고 있는 tmpfs/shared memory 경로
핵심: Pod가 PID를 가진다기보다, Pod 안의 컨테이너 프로세스들이 Node에서 PID를 가집니다. 따라서 접근할 때는 대상 Pod의 컨테이너 프로세스 PID를 찾아야 하며, Pod 재시작 또는 컨테이너 재생성 시 이 PID는 바뀝니다.

3방식 1: privileged Pod B가 /host/proc에서 동적 접근

시스템 구조 그림

동작 흐름

  1. Pod B 실행
  2. Pod B가 Kubernetes API로 Pod A 조회
  3. Pod A의 containerID 확인
  4. Pod B가 /host/proc/*/cgroup 검색
  5. Pod A의 host PID 확인
  6. /host/proc/<PID>/root/dev/shm/<ringbuffer> 접근
  7. 필요하면 /dev/shm/<ringbuffer>로 symlink 생성
  8. 기존 consumer처럼 파일 read
  9. Pod A 재시작 시 PID 재탐색

핵심 접근 경로

/host/proc/<PodA_PID>/root/dev/shm/<ringbuffer-file>

기존 consumer가 /dev/shm/<file>만 읽는 경우

디렉터리 전체가 아니라 파일만 symlink 처리하는 것이 안전합니다.

ln -sfn \
  /host/proc/${PID}/root/dev/shm/<ringbuffer-file> \
  /dev/shm/<ringbuffer-file>

Kubernetes 설정 예시

apiVersion: v1
kind: Pod
metadata:
  name: pod-b-privileged-reader
spec:
  hostPID: true

  containers:
    - name: consumer
      image: your-consumer-image:latest
      securityContext:
        privileged: true
        runAsUser: 0
      volumeMounts:
        - name: host-proc
          mountPath: /host/proc
          readOnly: true

  volumes:
    - name: host-proc
      hostPath:
        path: /proc
        type: Directory

장점

장점 설명
Pod A 수정 불필요 기존 Pod A 이미지, 프로세스, Helm chart를 건드리지 않음
Pod A 재시작 대응 가능 Pod B가 PID를 다시 찾으면 됨
자동화 가능 Pod B 내부 로직으로 복구 가능
네트워크 오버헤드 없음 TCP/gRPC/WebSocket 없이 Node-local 접근
shared memory/tmpfs 장점 유지 /dev/shm의 메모리 기반 접근 유지

단점

단점 설명
권한이 강함 privileged, hostPID, host /proc mount 필요 가능성이 큼
보안 승인이 어려울 수 있음 운영/보안 정책에서 막힐 수 있음
구현 복잡도 높음 PID 탐색, 재탐색, 예외 처리가 필요
같은 Node 제약 Pod A와 Pod B가 반드시 같은 Node여야 함
런타임 의존성 cgroup 경로가 containerd/docker/cri-o에 따라 다를 수 있음

문제점 / 장애 포인트

문제 원인 대응
PID를 못 찾음 containerID 불일치, cgroup 구조 차이 containerID prefix 제거, /host/proc/*/cgroup 검색 로직 보강
/dev/shm 파일이 안 보임 pause container PID를 잡았거나 잘못된 container PID 사용 target containerName 명확히 지정
Permission denied 권한/SELinux/AppArmor/seccomp 제한 privileged, runAsUser: 0, 보안 정책 확인
Pod A 재시작 후 경로 깨짐 PID 변경 PID 재탐색 후 symlink 갱신
frame 깨짐 링버퍼 동기화 규칙 미준수 기존 consumer의 read protocol 그대로 사용
보안상 주의: 이 방식은 privileged: true, hostPID: true, /proc hostPath mount를 사용하므로 Node 내부 정보 접근 범위가 큽니다. 운영 반영 전 보안 정책 확인이 필요합니다.

4방식 2: 외부에서 PID 확인 후 특정 shm 경로를 hostPath로 mount

시스템 구조 그림

동작 흐름

  1. Pod A는 이미 실행 중
  2. 운영자 또는 외부 스크립트가 Pod A의 containerID/PID 확인
  3. /proc/<PID>/root/dev/shm이 존재하는지 host에서 확인
  4. Pod B YAML 또는 Helm values에 hostPath로 해당 경로 지정
  5. Pod B 실행
  6. Pod B consumer는 /pod-a-shm/<ringbuffer>를 일반 파일처럼 읽음
  7. Pod A 재시작 시 운영자가 PID 재확인 후 Pod B 재배포

Kubernetes 설정 예시

apiVersion: v1
kind: Pod
metadata:
  name: pod-b-hostpath-reader
spec:
  nodeName: <POD_A_NODE_NAME>
  restartPolicy: Never

  containers:
    - name: consumer
      image: your-consumer-image:latest
      securityContext:
        privileged: false
        runAsUser: 0
      volumeMounts:
        - name: pod-a-shm
          mountPath: /pod-a-shm
          readOnly: true

  volumes:
    - name: pod-a-shm
      hostPath:
        path: /proc/<POD_A_PID>/root/dev/shm
        type: Directory
권장: Pod B 내부 mountPath는 /dev/shm 전체보다 /pod-a-shm 같은 별도 경로가 안전합니다. 기존 consumer가 /dev/shm 고정이라면 파일 symlink를 검토합니다.

실제 검증 결과: 직접 hostPath mount 실패

/proc/<PID>/root/dev/shm 경로 자체는 host에서 접근 가능한 경로처럼 보이지만, 이 경로를 Kubernetes hostPath로 Pod B에 직접 마운트하는 방식은 컨테이너 생성 단계에서 실패할 수 있습니다. 실제 테스트에서는 dpp-shm-test Pod 생성 중 runc mount 단계에서 다음 유형의 오류가 발생했습니다. 

Error: failed to create containerd task: failed to create shim task:
OCI runtime create failed: runc create failed:
error mounting "/proc/67837/root/dev/shm" to rootfs at "/dpp/shm":
mount /proc/67837/root/dev/shm:/dpp/shm (...), flags: 0x5001:
invalid argument

이 실패는 /proc/<PID>/root/dev/shm가 일반적인 정적 host 디렉터리와 성격이 다르기 때문에 발생한 것으로 볼 수 있습니다. 이 경로는 특정 프로세스의 root와 mount namespace를 통해 해석되는 procfs 기반 경로입니다. Kubernetes와 container runtime은 Pod를 만들 때 hostPath를 새 컨테이너 rootfs 안으로 bind mount해야 하는데, 대상이 procfs 경유 경로이고 그 안쪽의 /dev/shm가 tmpfs/mount namespace와 연결되어 있으면 runc가 bind mount 대상으로 안정적으로 처리하지 못할 수 있습니다.

현재 결론: 방식 2는 개념 설명과 후보안으로는 의미가 있지만, 실제 테스트에서 직접 mount가 실패했으므로 현재 구현 방향에서는 제외합니다. 대신 Pod B를 privileged: true로 실행하고 host /proc를 마운트한 뒤, 컨테이너 내부에서 /host/proc/<PID>/root/dev/shm을 직접 읽는 방식으로 진행합니다.

장점

장점 설명
Pod B가 privileged 없이 가능할 수 있음 이미 mount된 경로만 읽으므로 권한 축소 가능
Pod B 코드 단순 PID 탐색 로직 불필요
PoC가 빠름 host에서 PID만 확인하면 바로 테스트 가능
shared memory/tmpfs 장점 유지 /dev/shm의 메모리 기반 접근 유지
Pod A 수정 불필요 기존 시스템 영향 최소

단점

단점 설명
Pod A PID에 강하게 종속 Pod A 재시작 시 깨짐
자동 복구 어려움 Pod B가 스스로 새 PID를 모름
Node 이동에 취약 Pod A가 다른 Node로 가면 경로가 없음
manifest가 런타임 상태에 의존 선언형 Kubernetes 운영과 맞지 않음
수동 절차 필요 PID 확인 → values 갱신 → 재배포

문제점 / 장애 포인트

문제 원인 대응
Pod B가 Pending/ContainerCreating hostPath 경로가 해당 Node에 없음 nodeName을 Pod A Node로 고정
/pod-a-shm이 비어 있음 잘못된 PID 또는 잘못된 container PID containerName 기준 PID 재확인
Pod A 재시작 후 read 실패 PID 변경 Pod B 재배포
보안 정책 차단 hostPath 금지 namespace/admission 정책 확인
Permission denied shm 파일 권한 문제 runAsUser: 0 또는 권한 조정 필요
invalid argument /proc/<PID>/root/dev/shm 직접 hostPath mount를 runc가 처리하지 못함 방식 2 중단, 방식 1처럼 host /proc 전체를 mount한 뒤 런타임에 경로 접근
운영 주의: 방식 2는 Pod A가 거의 재시작되지 않는 환경에서도 직접 hostPath mount 단계가 통과해야만 사용할 수 있습니다. 현재 테스트 결과로는 runc mount 실패가 확인되었으므로 운영 후보에서 제외하고, 장애 기록으로 유지합니다.

7추천 진행안

빠른 PoC라면

방식 2 직접 mount는 보류: 실제 테스트에서 /proc/<PID>/root/dev/shm을 Pod B에 hostPath로 직접 mount할 때 runc invalid argument 오류가 발생했습니다.

운영 자동화까지 고려한다면

현재 추천: Pod B를 privileged: true로 실행하고 host /proc를 /host/proc로 마운트합니다. Pod B가 대상 PID를 찾은 뒤 /host/proc/<PID>/root/dev/shm 경로를 직접 읽는 방식으로 진행합니다.

단계별 진행

  1. host에서 직접 확인: /proc/<PID>/root/dev/shm 접근 가능 여부 확인
  2. 직접 hostPath mount 결과 기록: runc invalid argument로 실패한 케이스를 방식 2 한계로 유지
  3. 방식 1 PoC: privileged Pod B에 host /proc를 mount하고 런타임에 PID 경로 접근
  4. consumer 테스트: 100ms 주기로 bin 생성, CPU, latency, frame 유실 확인
  5. Pod A 재시작 테스트: PID 변경 시 Pod B의 PID 재탐색, symlink 갱신 또는 직접 경로 read 확인

최종 판단표

상황 추천
일단 되는지 빠르게 확인 방식 1: privileged Pod B + host /proc mount
Pod B 권한을 줄이고 싶음 방식 2가 후보였으나 직접 mount 실패로 현재는 보류

8Codex 요청용 요약

아래 내용을 Codex에 그대로 넘겨 작업 요청할 수 있습니다.

기존 Helm chart에서 동작 중인 Pod A는 수정하지 않고, 별도 Pod B를 추가하여 Pod A의 /dev/shm 링버퍼 파일을 읽는 구조를 검토하고 있다.

접근 방식은 2가지로 정리한다.

[방식 1: privileged Pod B 동적 접근]
- Pod B에 host /proc를 /host/proc로 readOnly hostPath mount한다.
- Pod B는 privileged: true, hostPID: true, runAsUser: 0으로 실행한다.
- Pod B가 Kubernetes API로 Pod A containerID를 조회한다.
- /host/proc/*/cgroup에서 containerID를 검색해 Pod A host PID를 찾는다.
- /host/proc/<PID>/root/dev/shm/<ringbuffer-file> 경로를 직접 읽는다.
- 기존 consumer가 /dev/shm/<file> 하드코딩이면 해당 파일만 symlink한다.
- Pod A 재시작 시 PID를 재탐색하고 symlink를 갱신한다.
- 장점: 자동화 가능, Pod A 재시작 대응 가능.
- 단점: privileged, hostPID, /proc mount로 보안 리스크가 큼.

[방식 2: PID 주입 hostPath mount]
- 외부에서 Pod A Node와 host PID를 먼저 확인한다.
- Pod B manifest 또는 Helm values에 /proc/<PID>/root/dev/shm을 hostPath로 지정한다.
- Pod B에는 /pod-a-shm으로 mount한다.
- Pod B consumer는 /pod-a-shm/<ringbuffer-file>을 일반 파일처럼 읽는다.
- 개념상 privileged: false, hostPID: false로도 가능할 수 있어 보였다.
- 하지만 실제 테스트에서 runc가 /proc/<PID>/root/dev/shm 직접 mount를 처리하지 못하고 invalid argument로 실패했다.
- Pod A 재시작 시 PID가 바뀌므로 Pod B를 재배포해야 한다.
- 장점: 성공한다면 Pod B 권한을 줄일 수 있음.
- 단점: 현재 환경에서는 직접 hostPath mount 실패로 구현 방향에서 제외.

공통 조건:
- Pod A와 Pod B는 반드시 같은 Node에 있어야 한다.
- /dev/shm이 tmpfs/shared memory 기반이라면 shared memory의 성능 장점은 대부분 유지된다.
- 단, 완전한 zero-copy는 아니며 Pod B가 read할 때 user-space copy는 발생한다.
- 링버퍼의 read pointer, sequence, partial frame 방지 등 기존 consumer의 read protocol을 그대로 지켜야 한다.
- output bin 파일은 .tmp로 쓴 후 rename 처리한다.

현재는 방식 2 직접 mount 실패를 장애 기록으로 남기고, 방식 1처럼 privileged Pod B에 host /proc를 마운트한 뒤 런타임에 /host/proc/<PID>/root/dev/shm을 직접 읽는 방향을 추천한다.
