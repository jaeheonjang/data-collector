Pod 분리 방안 도출 과정

Pod A를 그대로 유지하면서 consumer를 별도 작업 공간으로 분리하기 위해 검토한 대안, 제외 사유, 그리고 최종적으로 host /proc mount와 /proc/<PID>/root/dev/shm 접근 방식으로 정리된 이유를 기록합니다.

작성일: 2026-06-16 목적: Pod A 유지 + consumer 분리 현재 방향: privileged Pod + host /proc mount
1. 목적 2. 초기 대안 3. proc 접근 검토 4. 최종 판단

1. 목적과 제약

Pod A 유지

기존 Pod A의 Helm chart, container 구성, writer 동작을 최대한 변경하지 않습니다.

리소스 분리

Pod A 내부 consumer가 CPU 등 리소스를 사용하지 않도록 별도 Pod에서 처리합니다.

실시간 shm read

/dev/shm에 작성되는 ringbuffer 데이터를 지연과 누락을 최소화해 읽어야 합니다.

조건 설계에 주는 영향
Pod A는 그대로 유지 sidecar 추가, volumeMount 추가, writer 코드 변경 같은 선택지는 우선순위가 낮아집니다.
별도 작업 공간 필요 consumer는 Pod A 내부가 아니라 분리된 Pod에서 동작해야 합니다.
Pod A 리소스 보존 consumer 처리 비용을 Pod A의 CPU/메모리 budget에서 제거해야 합니다.
100ms 단위 ringbuffer 데이터 기존 consumer의 read protocol, sequence, partial frame 방지 규칙을 유지해야 합니다.

2. 초기 대안과 제외 사유

Sidecar container 방식

처음에는 Kubernetes 관점에서 가장 자연스러운 sidecar 방식이 제안되었습니다. writer container와 consumer container를 같은 Pod 안에 두고, memory emptyDir 또는 같은 Pod 내부 공유 경로를 통해 데이터를 읽는 구조입니다. 

Pod A
  writer container
    /dev/shm 또는 memory volume에 ringbuffer write

  consumer sidecar container
    같은 Pod 내부에서 ringbuffer read
장점: 같은 Pod 내부 공유는 Kubernetes에서 가장 정석적인 형태입니다. scheduling, lifecycle, tmpfs volume 공유가 명확하고 network hop이 없습니다.
제외 이유: Pod A spec 수정이 필요하고, consumer가 여전히 Pod A와 같은 Pod 리소스 범위에서 동작합니다. 이번 목표인 “Pod A를 그대로 유지”와 “Pod A 리소스를 온전히 사용” 조건에 맞지 않았습니다.

Pod 간 전달 계층

대안 장점 보류 이유
gRPC streaming Pod 간 실시간 전달 구조로 자연스럽습니다. Pod A writer 또는 producer 코드 변경이 필요합니다.
NATS JetStream ack, replay, durable consumer를 제공할 수 있습니다. 별도 메시징 인프라와 publish 로직이 필요합니다.
Kafka 장기 보존, 재처리, consumer group에 강합니다. 현재 요구에는 운영 복잡도가 큽니다.
Redis 최신 데이터 공유나 pub/sub 구현이 단순합니다. 모든 chunk 보존, 순서 보장, 재처리에 약합니다.
PVC/RWX 파일 기반 계약에는 맞습니다. shm ringbuffer 실시간 read 목적과 다르고 partial write 방지 정책이 필요합니다.

3. proc 기반 접근으로 좁혀진 과정

Pod A를 수정하지 않는 조건 확정

sidecar와 memory volume 공유는 정석적이지만 Pod A spec 변경이 필요해 제외했습니다.

별도 Pod에서 shm을 직접 읽는 요구 확인

consumer를 분리하되, ringbuffer 데이터는 실시간으로 받아야 하므로 Pod A 내부 파일 생성 이후의 배치 전달만으로는 부족했습니다.

/proc/<PID>/root/dev/shm 접근 검토

Pod A container의 host PID를 기준으로, 해당 프로세스가 보는 root filesystem을 따라 들어가 /dev/shm을 읽는 방식을 검토했습니다.

직접 hostPath mount 시도

권한을 줄이기 위해 /proc/<PID>/root/dev/shm만 Pod B에 mount하는 방식을 시도했지만 runc invalid argument로 실패했습니다.

host /proc mount 방식으로 정리

Pod B를 privileged로 실행하고 host /proc를 read-only mount한 뒤, runtime에 대상 PID 경로를 직접 읽는 방식으로 정리했습니다.

/proc/<PID>/root/dev/shm의 의미

Kubernetes container는 Node 입장에서는 Linux process입니다. host의 /proc/<PID>/root는 해당 process가 container 내부에서 /로 보는 root filesystem을 가리킵니다. 따라서 /proc/<PID>/root/dev/shm은 Pod A container 내부의 /dev/shm을 host에서 따라 들어가는 경로입니다.

직접 hostPath mount 실패

Error: failed to create containerd task: failed to create shim task:
OCI runtime create failed: runc create failed:
error mounting "/proc/<PID>/root/dev/shm" to rootfs at "/dpp/shm":
mount /proc/<PID>/root/dev/shm:/dpp/shm (...), flags: 0x5001:
invalid argument
판단: /proc/<PID>/root/dev/shm은 일반적인 정적 host directory가 아니라 특정 process의 root와 mount namespace를 통해 해석되는 procfs 경로입니다. 직접 hostPath mount는 현재 환경에서 실패했으므로 구현 후보에서 제외했습니다.

4. 최종 판단

항목 판단
sidecar container Kubernetes 정석에 가깝지만 Pod A 유지와 리소스 분리 목적에 맞지 않아 제외
gRPC/NATS/Kafka/Redis/PVC 장기적으로는 표준적이나 Pod A 변경 또는 별도 인프라가 필요해 즉시 적용 제외
직접 hostPath mount 권한 축소 측면에서 매력적이었으나 runc invalid argument로 실패
privileged Pod + host /proc mount 현재 조건에서 Pod A를 유지하고 consumer를 별도 Pod로 분리하기 위한 현실적인 선택
현재 방향: Pod B를 privileged: true로 실행하고 host /proc를 read-only mount합니다. Pod B가 Pod A container PID를 찾은 뒤 /host/proc/<PID>/root/dev/shm 경로에서 ringbuffer 데이터를 읽습니다.

Confluence용 요약

Pod A를 변경하지 않고 consumer를 별도 작업 공간으로 분리하기 위해 여러 대안을 검토했다. 초기에는 Kubernetes에서 자연스러운 sidecar container와 memory emptyDir 공유 방식이 제안되었지만, 이는 Pod A spec 변경이 필요하고 consumer가 여전히 Pod A의 리소스 범위 안에서 동작하므로 “Pod A를 그대로 유지하고 리소스를 온전히 사용하게 한다”는 목적과 맞지 않았다. gRPC, NATS, Kafka, Redis, PVC 같은 Pod 간 전달 구조도 검토했지만 Pod A 변경 또는 별도 인프라 추가가 필요했다. 이후 Pod A container의 host PID를 기준으로 /proc/<PID>/root/dev/shm에 접근하는 방식을 검토했고, 직접 hostPath mount는 runc invalid argument로 실패했다. 최종적으로는 별도 Pod를 privileged로 실행하고 host /proc를 read-only mount한 뒤, runtime에 Pod A PID를 찾아 /host/proc/<PID>/root/dev/shm 경로에서 ringbuffer 데이터를 읽는 방향으로 정리했다.
